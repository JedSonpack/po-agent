# s10–s15 开发总览（meta-plan）

> 逐阶段仍走 superpowers 流程（设计规格 → 实现计划 → TDD → commit → PROGRESS）。本文件是 6 阶段的总体方案与架构对齐决策。

## 已确认决策
- **推送**：每阶段末 `git push origin/main`；每任务本地 commit（`feat(sNN)`/`docs(sNN)`/`chore(sNN)`/`fix(sNN)`）。
- **节奏**：s10→s15 连续做完，阶段末自然停顿。
- **架构**：沿用 po-agent 既定模式 —— **累积式**（保留上阶段全部机制）+ **DI**（`agent_loop` 依赖注入，无模块级 client/model 全局）+ **TDD**（mock 化，不发真实 API）。参考 `learn-claude-code/sNN` 是**行为基准**，结构重构。

## 跨阶段架构对齐（关键）
参考 s10-s15 用 `agent_loop(messages, context)` + 模块全局，且为教学聚焦**丢弃**前序机制（s12 丢 s11 恢复；s10-s13 只 3 工具）。po-agent **不跟随简化**：每阶段复制上阶段整包（sed 改名）再叠加一个新机制，循环签名随 DI 演进。到 s15 将累积：hooks/权限 + TodoNag + Compactor + Subagent + skills + Memory + system_prompt 组装 + recovery + tasks + background + cron + teams。

## 各阶段

### s10 System Prompt — 运行时组装系统提示
- 新模块 `system_prompt.py`：`PROMPT_SECTIONS`（identity/tools/workspace/skills/memory 段）+ `assemble_system_prompt(context)` + `get_system_prompt(context)`（`json.dumps(sort_keys=True)` 做 cache key，命中打 `[cache hit]`/`[assembled] sections: ...`）+ `build_context(*, cwd, tools, skills_catalog, memories)`。
- **agent_loop 重构**：drop `system: str` 入参 → 改取 `context: dict`；循环内每轮重算 `context`（重读 memory 索引）+ `system = get_system_prompt(context)`。把 s09 的 `sys_prompt = system + memory.build_index_section()` 收进 `assemble_system_prompt`。
- config.py：`build_system_prompt` → `build_context`；`load()` 返回 context 组件。
- 保留 s09 全部（hooks/nag/compact/memory/skills/subagent/9 工具）。**无新工具**。
- 测试：assemble 3 段 vs 含 memory/skills 多段、cache hit/miss、build_context 派生、agent_loop 用 context（更新 s09 复制的 test_agent：`system=`→`context=`）。
- 风险：agent_loop 签名变化，s09 复制的 test_agent 需改。s11+ 复制 s10 时已是 context 版，无额外改动。

### s11 Error Recovery — 错误重试与恢复
- 新模块 `recovery.py`：`RecoveryState`（has_escalated/recovery_count/consecutive_529/has_attempted_reactive_compact/current_model）+ `with_retry(fn, state)`（429/529 指数退避，529 连续 3 次切 fallback model）+ `retry_delay(attempt)` + `is_prompt_too_long_error(e)` + 常量（DEFAULT_MAX_TOKENS=8000、ESCALATED_MAX_TOKENS、MAX_RETRIES=10、MAX_CONSECUTIVE_529=3、CONTINUATION_PROMPT）。
- **agent_loop 集成**：LLM 调用包 `with_retry`；`stop_reason=="max_tokens"` → 升级 max_tokens（1 次，不 append 截断输出）→ 续写提示（最多 3 次）；外层 except：prompt_too_long → reactive_compact（限 1 次，用 `RecoveryState.has_attempted_reactive_compact`）。
- 复用 s08 `Compactor.is_prompt_too_long` / `reactive_compact`（比参考"留最后 5 条"更丰富，po-agent 保留）；参考 standalone reactive_compact 不引入。
- 保留 s10 全部。**无新工具**。
- 风险：glm-5.2 推理模型，ESCALATED_MAX_TOKENS（参考 64000）可能超 ARK 端点上限 —— 设为可配置常量，冒烟验证；续写/升级对 thinking 模型的效果冒烟确认。

### s12 Task System — 可追踪任务（DAG）
- 新模块 `tasks.py`：`Task` dataclass（id/subject/description/status/owner/blockedBy）+ `.tasks/{id}.json` 持久化 + `create_task/save_task/load_task/list_tasks/get_task/can_start/claim_task/complete_task` + 5 工具 handler。
- 工具进 `TOOL_HANDLERS` + `make_tools()`（共 14 工具）。
- `can_start`：blockedBy 全 completed 才 True；缺失依赖=blocked 不抛。`complete_task` 完成后扫描报 unblocked 下游（不自动 claim）。
- agent_loop 不主动咨询任务系统（逻辑全在 handler）—— 与参考一致。
- 保留 s11 全部（含 recovery）。
- 测试：create/persist/list/get、can_start（缺失/未完成/完成/多依赖）、claim（状态机+blocked+owner）、complete（unblocked 报告/无下游）、DAG 端到端、跨会话持久。`tmp_path` 隔离 TASKS_DIR。

### s13 Background Tasks — 慢操作后台执行
- 新模块 `background.py`：`background_tasks`/`background_results`/`background_lock`/`_bg_counter` + `is_slow_operation`（bash 关键词启发）+ `should_run_background`（`run_in_background` 显式优先）+ `start_background_task(block)`（daemon 线程）+ `collect_background_results()`（pop completed → `<task_notification>`，summary 截 200）。
- bash schema 加 `run_in_background: boolean`（`run_bash` 内忽略，dispatch 层判断）。
- **agent_loop 集成**：tool_use block 先判 `should_run_background` → 后台派发 + 占位 tool_result；否则 `run_tool` 同步。构造 user 消息前 `collect_background_results()` → 通知作 text block 追加（**results 在前、通知在后**，以 code.py 为准）。
- 提取 `execute_tool` 复用（worker 用）。worker 加 try/except（po-agent 改进：异常标 completed 写错误，避免参考的静默泄漏）。
- 保留 s12 全部。
- 测试：is_slow_operation/should_run_background、start_background_task 派发+生命周期+id 自增、collect 格式+截断+pop+只收 completed、agent_loop 后台占位+通知注入顺序、同步路径不变、stop_reason 非 tool_use 不收集。

### s14 Cron Scheduler — 按时间表触发
- 新模块 `cron.py`：`CronJob` dataclass（id/cron/prompt/recurring/durable）+ `cron_matches`/`_cron_field_matches`（5 字段，DOM/DOW OR 语义，DOW 换算 `(weekday+1)%7`）+ `validate_cron`/`_validate_cron_field` + `schedule_job`/`cancel_job` + `cron_queue`/`cron_lock`/`_last_fired` + `cron_scheduler_loop`（daemon，每秒轮询，`minute_marker="%Y-%m-%d %H:%M"` 去重）+ `consume_cron_queue`/`has_cron_queue` + `queue_processor_loop` + `agent_lock` + `save_durable_jobs`/`load_durable_jobs`（`.scheduled_tasks.json`）+ 3 工具（schedule_cron/list_crons/cancel_cron）。
- **线程启动显式化**（po-agent 改进）：参考在 import 时起调度线程（不可测）；po-agent 把 `start_scheduler()` 放 cli main 调用，import 不起线程。
- **agent_loop 集成**：循环顶部 `consume_cron_queue()` → 每个 fired job 注入 `[Scheduled] {prompt}` user 消息。
- 保留 s13 全部（background 通知 + cron 注入同 loop 协同）。
- 测试：cron_matches（`*`/`*/N`/`N-M`/`N,M`/DOM-DOW OR/DOW 换算/非 5 字段）、validate_cron（越界/步长 0/range 反/字段数/非数字）、schedule/cancel、save/load 往返（含非法跳过）、consume/has_queue、agent_loop 注入、queue_processor 协调（mock agent_lock）。`cron_scheduler_loop` 用可注入时间 + 短 sleep 测去重与 one-shot 删除。

### s15 Agent Teams — 多 agent 组队
- 新模块 `teams.py`：`MessageBus`（`.mailboxes/{agent}.jsonl`，`send` append / `read_inbox` 消费式 read+unlink / `peek` 非消费）+ `spawn_teammate_thread(name, role, prompt)`（daemon 线程跑简化 loop：4 工具 bash/read_file/write_file/send_message，max 10 轮，每轮顶 `read_inbox(name)` 注入 `<inbox>`，sliding window `messages[-20:]`，完成发 summary 给 lead + pop active_teammates）+ `active_teammates` + 3 lead 工具（spawn_teammate/send_message/check_inbox）。
- teammate 的 send_message `from`=teammate name（非 lead）；teammate 无 spawn_teammate（防递归组队）。
- **cli 重构为事件队列**（po-agent 改进：显式起 input_reader/inbox_poller daemon）：events queue；inbox_poller 用 `BUS.peek("lead")`（非消费）触发 wake；wake 注入 `[Inbox]` + background 通知；"all teammates done" 通知。
- 与 s06 Subagent 区别：s06 同步一次性回总结；s15 异步 daemon + 文件邮箱多轮。
- 保留 s14 全部。
- 测试：MessageBus send/read/peek/消费性/跨 agent 隔离、spawn 去重+注册、run_send_message(from=lead)、run_check_inbox(空/格式/消费/截断)、teammate run()（mock client：max 10 轮/inbox 注入/sliding window/summary 提取/完成发 result+pop）、wake 注入格式+幂等、"all done" 条件。read+unlink 竞态已知，po-agent 用锁改进或标注。

## 每阶段工作流（superpowers）
1. 设计规格 `docs/superpowers/specs/2026-07-04-sNN-*-design.md`
2. 实现计划 `docs/superpowers/plans/2026-07-04-sNN-*.md`（任务级 checkbox + 测试代码）
3. TDD 执行：包骨架 → 复制上阶段模块（sed 改名）→ 新模块（先失败测试→实现）→ agent_loop 集成 → config/cli → README → 全测 → 冒烟 → PROGRESS
4. 每任务一 commit；阶段末 push origin/main
5. 更新 PROGRESS.md（行 ⬜→🚧→✅ + 「已完成阶段详情」一节）

## 验收
- 每阶段：全量 s01-sNN 测试通过 + 实时冒烟（`echo '...' | python -m sNN_*`）跑通。
- 累积到 s15：所有机制共存可跑。

## 风险
- **glm-5.2 推理模型**：s11 max_tokens 升级值需冒烟验证 ARK 端点上限；任何 auxiliary LLM 调用需大 max_tokens（≥2000，thinking 吃 token）。
- **s14/s15 线程**：daemon 线程测试受限，聚焦纯函数（cron 匹配/校验/队列、MessageBus）+ mock client；线程启动显式化避免 import 副作用。
- **s10 签名变化传导**：仅 s10 复制 s09 时需改 test_agent（`system=`→`context=`）；s11+ 复制 s10 已是 context 版。
